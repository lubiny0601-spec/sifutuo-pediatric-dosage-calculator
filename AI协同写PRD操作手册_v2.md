# 如何与 AI 一步步对话，生成你的工具 PRD

**适用工具：** ChatGPT / Gemini / Claude  
**适用对象：** 无产品经理或开发经验的市场、销售同事  
**预计时长：** 30–45 分钟完成全部对话，得到可交付的 PRD 文档

---

## 开始之前，先明白三件事

> **① 你只需要做判断，不需要原创内容**
> AI 会帮你把结构写出来，你只需要说「这个对 / 这个改一下 / 这个不要」。
>
> **② 每一轮只解决一个问题**
> 不要一次性把所有问题说给 AI，一轮一轮来，每轮都有明确的产出物。
>
> **③ AI 不知道的业务特殊性，必须你来说**
> 比如「需要 Medical & Legal 审批」「数据只能用说明书」这类内部约束，AI 猜不到，必须你主动告知。

---

## 第 1 轮：描述业务痛点

### 你的目标
让 AI 理解你的处境，让它来问你——不是你告诉它解决方案。

### 你发给 AI 的 Prompt

```
我是辉瑞的医药代表，负责推广抗感染产品 CAZ-AVI 和 SUL-DUR。
拜访 ICU 医生时经常被问到肾功能不全的患者该怎么调整剂量，
我当场查不到，显得很不专业。

请先问我几个问题帮我搞清楚真正的需求，不要急着给我解决方案。
```

> **关键技巧：** 第一轮只说痛点，不要说「我要做一个什么功能」。你说得越像在倒苦水，AI 问出来的问题就越有价值。

### AI 可能会这样回复

```
好的，我先问几个问题：

① 这种情况大概多频繁？每周几次？
② 你现在是怎么处理的？说「稍后回复您」，
   还是当场翻说明书？
③ 有多少同事也遇到同样的问题？
④ 拜访时你主要用手机还是平板？
```

### 你需要做什么
如实回答 AI 的每一个问题，说具体数字，不要说「有时候」「偶尔」。

### ✅ 本轮结束的标志
AI 整理了你的回答，说「我的理解是……这准确吗？」

---

## 第 2 轮：确认用户画像

### 你的目标
让 AI 帮你整理出「谁在用这个工具、在什么场景下用、不给谁用」。

### 你发给 AI 的 Prompt

```
你的理解准确。现在请帮我整理出目标用户画像，
包括：使用者是谁、在什么场景用、用什么设备、
他们最核心的诉求是什么、哪些人不适合用这个工具。
```

### AI 可能会这样回复

```
根据你的描述，我整理出以下用户画像：

| 维度       | 描述                                     |
|----------|----------------------------------------|
| 用户角色   | 辉瑞抗感染事业部医药代表（约 8 人）         |
| 专业背景   | 医药销售背景，非临床专业，不应自行换算剂量   |
| 使用场景   | ICU / 感染科拜访中，手机端实时查询          |
| 触发时机   | 医生主动提问，或介绍产品时涉及肾功能场景     |
| 核心诉求   | 快（30 秒出结果）、准（来自说明书）、简（步骤少）|
| 暂不开放   | 医生、护士、患者及任何外部用户              |

这个理解准确吗？有需要调整的地方吗？
```

### 你需要做什么
检查每一行是否符合实际，有不对的直接说「XX 那条改成……」

### ✅ 本轮结束的标志
用户画像内容你已逐行确认。

---

## 第 3 轮：功能取舍

### 你的目标
让 AI 列出所有可能的功能，你来决定哪些现在做、哪些以后做、哪些不做。

### 你发给 AI 的 Prompt

```
请帮我列出这个工具可能需要的所有功能，
然后按「现在就做 / 以后再做 / 暂时不做」分三类，
我来做最终决定。
```

> **关键技巧：** 用「现在就做 / 以后再做 / 暂时不做」这三个词，AI 会自动按产品经理的优先级方法帮你分类，不需要你懂这个方法论。

### AI 可能会这样回复

```
根据你的场景，我来分类整理：

【现在就做】
- 免责声明确认（首屏，用户点击后方可继续）
- 药品选择（CAZ-AVI / SUL-DUR）
- 肾功能数值输入
- eGFR 分级快速选择按钮
- 剂量结果卡片展示
- 说明书来源标注
- 一键复制备忘

【以后再做】
- 收藏常用查询
- 本地查询记录
- 更多药品扩展

【暂时不做】
- 儿科剂量
- 连接医院 HIS 系统
- 用户登录账号
- AI 自行推断剂量
- 面向医生或患者开放

你对这个分类有什么调整吗？
```

### 你需要做什么
- 逐条确认「现在就做」的功能，有补充直接说「还要加上 XX 功能」
- 「暂时不做」里每条，说清楚不做的原因（AI 会帮你写进 PRD）

**本案例需要补充的内容：**
- 加入「高风险场景提示」（AKI / CRRT / 透析不明确时触发提示，不输出剂量结论）

### ✅ 本轮结束的标志
三类功能清单你已逐条确认，AI 记录了你的修改。

---

## 第 4 轮：数据来源与合规

### 你的目标
明确 AI 依据什么资料回答、哪些内容必须经过审核才能上线。

> ⚠️ **为什么这一轮最容易被跳过，但最关键？**
> 很多人写完功能就以为工具可以上线了。但医药工具的数据来源和审批流程是上线的前置条件。跳过这步，技术同事做出来的东西可能根本过不了 Medical & Legal 审核。

### 你发给 AI 的 Prompt

```
这个工具里的剂量数据，AI 应该依据什么来回答？
如果数据错了，谁负责？
有哪些内容需要在 PRD 里写清楚？
```

### AI 可能会这样回复

```
这个问题非常关键，尤其在医药场景。我帮你拆解三个层次：

① 数据从哪里来？
  · 只用 CFDA 批准说明书（最安全，合规风险最低）
  · 说明书 + 权威指南
  · 允许 AI 根据知识库自行推断（风险最高，不建议）

② 谁来确认数据正确？
  发布前需 Medical 团队逐一核对，通过后才能上线，
  不能由技术同事自行录入。

③ 界面免责声明
  必须有「本工具仅供参考，临床给药决策由医生负责」
  的声明，且在首屏和结果页固定展示，不可关闭。

另外提醒：CAZ-AVI 和 SUL-DUR 在说明书中使用的肾功能
指标可能不同（eGFR vs CrCl），两个药品的分层阈值不得
强行统一，须逐药由 Medical 确认。

你们是否需要 Legal 审批才能发布？
```

### 你需要确认的关键点

| 问题 | 本案例的答案 | 你自己的答案 |
|---|---|---|
| 剂量数据来自哪里？ | 仅限 CFDA 批准说明书 | |
| AI 可以自行推断数字吗？ | 不可以，禁止 | |
| 发布前需要谁审核？ | Medical 逐条核对 | |
| 需要哪个部门审批？ | Medical + Legal 双重审批 | |
| 界面需要什么声明？ | 首屏 + 结果页固定显示 | |

### ✅ 本轮结束的标志
数据来源、审核流程、免责声明均已确认，AI 记录完毕。

---

## 第 5 轮：使用边界与 AI 角色

### 你的目标
明确工具「给谁用 / 不给谁用」，以及 AI 在工具里「能做什么 / 绝对不能做什么」。

### 你发给 AI 的 Prompt

```
除了合规，这个工具还有哪些使用边界需要在 PRD 里写清楚？
另外，AI 在这个工具里能做什么、不能做什么？
请把 AI 的角色边界整理出来，后面要写成一个独立章节。
```

### AI 可能会这样回复

```
使用边界：
· 仅限成人患者，界面标注「不适用于儿科」
· 仅供辉瑞代表内部使用，不对外开放
· 不存储任何患者信息，不连接医院系统
· 说明书更新后工具须同步更新，需指定负责人

AI 角色边界：

✅ AI 可以做：
- 引导用户补充必填信息（药品 / 肾功能指标）
- 识别输入是否完整、格式是否异常
- 将数据表内容组织成更易读的展示语言
- 高风险场景提示，引导转人工

❌ AI 不可以做：
- 自行生成、推断或换算任何剂量数字
- 换算不同肾功能指标（eGFR ↔ CrCl）
- 替代医生判断具体患者的给药方案
- 生成可直接外发给医生的医学建议
- 撰写未经 Legal 审核的免责声明
```

### 你需要做什么
逐条确认，并要求 AI 把「AI 角色边界」作为 PRD 的独立章节。

### ✅ 本轮结束的标志
使用边界和 AI 角色限制均已确认。

---

## 第 6 轮：输出完整 PRD

### 你的目标
让 AI 把前五轮所有内容整合成结构完整的 PRD 文档。

### 你发给 AI 的 Prompt

```
好，请把我们前面讨论的所有内容整理成一份完整 PRD。
要求：
- AI 角色边界单独成章
- 包含功能详细描述（每个功能写清楚用户操作、
  系统处理、输出结果、异常处理）
- 包含数据来源与合规要求章节
- 包含数据模版章节（说明模版的作用、由谁准备、
  字段结构、与工具输出的对应关系）
- 包含使用边界与高风险场景章节
- 包含验收标准和测试场景清单
- 格式清晰，我要交给技术同事参考实现
```

### AI 会输出的 PRD 章节结构

```
一、   背景与目标
二、   目标用户与使用场景
三、   功能范围（现在做 / 以后做 / 暂时不做）
四、   功能详细描述（F1 免责声明确认 · F2 药品选择 ·
       F3 肾功能输入双模式 · F4 剂量结果卡片 ·
       F5 来源查看 · F6 内部复制备忘）
五、   AI 角色边界
六、   典型使用流程
七、   数据来源与合规要求
八、   使用边界与高风险场景
九、   数据模版        ← 工具制作者需要提前准备
十、   验收标准与测试场景
附录   六轮对话法与章节对照
```

### 如何局部修改（不需要重新来过）

拿到 PRD 之后，如果有不对的地方，直接告诉 AI：

```
把验收标准里的「不超过 4 步」改成「不超过 5 步」
```

```
在数据来源表里加一行：
数据类型：eGFR/CrCl 指标匹配
允许来源：各药品说明书明确使用的指标
禁止来源：跨药品强行统一指标
```

### ✅ 本轮结束的标志
PRD 已输出，你已逐章检查并完成必要的局部修改。

## 最终产出物长什么样

### PRD 文档整体结构

一份完整的工具 PRD 大约包含以下十章内容。下面列出各章的核心内容，帮助你对照检查是否完整：

**第一章：背景与目标**
- 业务痛点描述（谁遇到了什么问题，频率是多少）
- 产品目标表格（G1 操作流畅 / G2 数据可追溯 / G3 降低门槛 / G4 控制合规风险）

**第二章：目标用户**
- 用户画像表格（角色、背景、场景、设备、诉求）
- 明确哪些人不在开放范围

**第三章：功能范围**
- 三列清单：现在就做 / 以后再做 / 暂时不做
- 每条「暂时不做」都附上原因

**第四章：功能详细描述**
每个功能用表格写清楚四个维度，以「剂量结果卡片」为例：

| 维度 | 内容 |
|---|---|
| 输出内容 | 单次剂量、给药频次、输注时间、特殊备注 |
| 视觉呈现 | 剂量数字最大，频次次之，备注灰色小字，5 秒内可读完 |
| 系统处理 | 从数据模版读取对应行；无记录时不生成结果，展示提示 |
| 来源标注 | 资料名称 + 版本日期 + 章节编号 + Medical 审核状态 |
| 高风险提示 | 透析 / AKI / CRRT 场景不展示剂量结论，提示转 Medical |

**第五章：AI 角色边界**
- 绿色列：AI 可以做的事（引导、识别、组织展示、提示）
- 红色列：AI 绝对不能做的事（推断数字、换算指标、替代医生判断）

**第六章：典型使用流程**
- 8 步操作路径表格，每步写明用户操作 + 界面反馈 + 体验目标

**第七章：数据来源与合规**
- 数据来源规则表（允许来源 vs 禁止来源）
- 数据表字段定义（14 个字段）
- 五环节审核流程（含负责方和产出物）
- 界面免责声明原文（待 Legal 审核确认）

**第八章：使用边界与高风险场景**
- 边界维度表（适用用户 / 患者人群 / 药品范围 / 数据存储 / 内容更新）
- 高风险场景处理表（7 个场景，每个场景写明工具的处理方式）

**第九章：数据模版**
- 模版的作用与准备责任（谁填、谁审、何时完成）
- 14 个字段的说明表格
- 数据模版与工具输出的映射关系（用户操作 → 读取哪个字段 → 展示什么）
- 维护规则（说明书更新 / 新增药品 / 发现错误 / 顾问变更）

**第十章：验收标准与测试场景**
- 8 条上线前验收标准（操作流畅 / 数据准确 / AI 边界 / 合规审批等）
- 9 个测试场景清单（正常 / 边界值 / 缺失信息 / 高风险 / 隐私输入等）

## 工具制作者需要提前准备的数据模版

> 数据模版是工具的「知识库」。用户输入 eGFR 值和药品名后，工具从模版中匹配对应行，读取剂量信息并展示。**AI 不自行推断任何数字，所有输出均来自模版中经 Medical 审核的内容。**

### 模版是什么

一份 Excel 文件，包含四个工作表：

| 工作表 | 内容 |
|---|---|
| 使用说明 | 文件用途、填写责任、维护规则 |
| 数据字典 | 14 个字段的含义和填写规则 |
| CAZ-AVI | 该药品所有肾功能档位的剂量数据 |
| SUL-DUR | 该药品所有肾功能档位的剂量数据 |

### 每一行数据包含的字段

每一行对应一个「药品 + 肾功能档位」的组合：

| # | 字段名 | 示例值 | 说明 |
|---|---|---|---|
| 1 | drug_name | CAZ-AVI | 药品名，与工具界面选项一致 |
| 2 | renal_indicator | CrCl | 该药品说明书使用的指标（eGFR 或 CrCl），**逐药确认，不可统一** |
| 3 | renal_range_label | 31–50 mL/min | 数值范围，严格按说明书表述 |
| 4 | renal_level_desc | 中度肾功能损伤 | 档位临床描述，界面辅助展示 |
| 5 | dose_single | （见说明书） | 单次剂量，来自说明书原文 |
| 6 | dose_frequency | （见说明书） | 给药频次，来自说明书原文 |
| 7 | infusion_duration | （见说明书） | 输注时间，无则留空 |
| 8 | special_note | | 备注（如负荷剂量提示），无则留空 |
| 9 | risk_flag | FALSE | **TRUE = 高风险，不输出剂量，触发提示** |
| 10 | source_chapter | §4.2 特殊人群用药 | 说明书章节，用于界面来源标注 |
| 11 | source_doc_version | 2024 年 3 月版 | 说明书版本日期 |
| 12 | medical_reviewed_by | 张医学顾问 | **Medical 审核人，空行不得上线** |
| 13 | medical_review_date | 2025-07-01 | Medical 审核日期 |
| 14 | valid_until | 2026-06-30 | 有效期，到期须复审 |

### 数据模版如何驱动工具输出

用户的每一个操作，都对应模版里特定字段的读取：

| 用户操作 | 工具读取的字段 | 用户看到什么 |
|---|---|---|
| 选择药品 | drug_name | 进入该药品查询界面 |
| 输入 eGFR 值或选档位 | renal_range_label + renal_indicator | 档位高亮；指标不匹配时提示确认 |
| 点击查询 | dose_single / dose_frequency / infusion_duration / risk_flag | risk_flag=FALSE：展示剂量卡片；risk_flag=TRUE：高风险提示，不出结论 |
| 点击查看来源 | source_chapter / source_doc_version / medical_reviewed_by / medical_review_date | 说明书章节 + 版本 + 审核人 + 审核日期 |
| 点击复制 | 以上字段组合 | 「[内部参考] 药品名（档位）：剂量 频次。来源：章节 版本。仅供内部参考。」 |
| 查询字段为空 | 无匹配行 | 「当前资料库无明确依据，请查阅完整说明书或咨询医学团队」 |

### 谁来准备，什么时候准备

| 环节 | 负责人 | 时间节点 |
|---|---|---|
| 填写数据内容 | 市场团队指定负责人 | 工具开发启动前完成初版 |
| 逐行医学审核 | Medical Advisor | 上线前完成，空行视为未审核 |
| 格式确认 | 技术同事 | 确认字段格式可被工具正确读取 |
| 发布审批 | Medical + Legal | 签署审批单，作为上线前置条件 |
| 说明书更新后复审 | 市场指定负责人 + Medical | 更新后 30 天内完成，期间相关查询暂停 |

## 六轮对话速查卡

| 轮次 | 你说的关键句 | 本轮产出 |
|---|---|---|
| 第 1 轮 | 「请先问我几个问题，不要急着给解决方案」 | AI 开始追问你 |
| 第 2 轮 | 「请整理出目标用户画像」 | 用户角色、场景、不开放人群 |
| 第 3 轮 | 「按现在就做 / 以后再做 / 暂时不做分类」 | 功能优先级，不做原因 |
| 第 4 轮 | 「数据依据什么？如果数据错了谁负责？」 | 来源规则、Medical 审核、Legal 审批 |
| 第 5 轮 | 「使用边界和 AI 角色边界各是什么？」 | 适用范围、AI 红线、高风险场景 |

## 常见问题

**Q：AI 给出的功能描述太简单，只有名称没有细节怎么办？**

在第 6 轮的 Prompt 里加上：

```
功能详细描述部分，每个功能必须包含：
用户操作、系统/AI 处理逻辑、输出结果、异常处理四个维度，
不能只写功能名称。
```

**Q：PRD 里没有数据模版章节怎么办？**

直接告诉 AI：

```
请新增一章「数据模版」，说明：
① 数据模版的作用（工具的知识库，AI 从中读取剂量数据）
② 由谁准备、谁来审核、什么时候准备
③ 模版的字段结构（14 个字段，逐一说明）
④ 用户操作与模版字段的映射关系
⑤ 模版的维护规则
```
